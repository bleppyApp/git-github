GIT Commands

Basic Commands

git status

copy

gives status on a git repository and its contents
git init

copy

creates a new repository for the current folder. Before initialize a repo, run git status first to make sure that the current location is not being watched by git.
git add

copy

stages changes before you can commit them. To stage multiple files at once, separate files with spaces, like: git add file1 file2
git add .

copy

stages all changes at once
git rm --cached fileName

copy

unstages a staged file

Commit

git commit -m "my message about the commitment"

copy

moves the staged changes from the staging area to the repository.
git commit -am "my message about the commitment"

copy

shortcut for staging and committing all files at once.
git commit --amend

copy

allows you to add a forgotten file(s) to the last commit or edit the message from the last commit. If the idea is to add a file, you'll have to stage it first with git add.

Branch

git branch

copy

gives the list of existing branches. And the one with an * is the active branch
git branch -v

copy

gives the list of existing branches, and their last commitment message.
git branch name_of_branch

copy

creates a new branch based on HEAD but it does not switch to it.
git branch -m newName

copy

modifies/renames the current branch
git branch -d name_of_branch

copy

copy

deletes branch. However, to make this work, this branch has be merged first. If you don't want to merge it, you can force the deletion by using uppercase d, -D, instead of -d. Remember to leave the branch before deleting it.

Merge

git merge branchName

copy

before merging, remember to switch to the master or the branch that you want the current branch to be merged into.

Diff

git diff

copy

shows the difference between the staging area (everything that was staged but not committed yet) and the working directory
git diff HEAD

copy

lists all changes in the working tree since your last commit, including staged and unstaged changes.
git diff --staged

copy

git diff --cached

copy

lists the difference between the staging area and the last commit
git diff (HEAD or --staged or nothing) fileName

copy

will show the differences inside the specific file
git diff branch1..branch2

copy

show differences between two branches. In this case, the order matters. branch1 will occupy the "old" or "a" or "---" and branch2 will be the "new" or "b" or "+++"
git diff commit1..commit2

copy

show differences between two commits. In this case, the order matters. commit1 will occupy the "old" or "a" or "---" and commit2 will be the "new" or "b" or "+++"

Stash

git stash

copy

cleans the working tree and saves staged and unstaged changes, so they don't move branches with you if you're don't want to or you're not ready to commit them.
git stash pop

copy

retrieves the last stashed changes to your current branch and cleans the stage.
git stash apply

copy

retrieves the stashed changes to your current branch BUT doesn't clean the stage, so you can use the same stash on multiple branches.
git stash list

copy

shows all saved stashes
git stash apply stash@{number}

copy

from stash list, you'll get the number of stash you wish to apply. This is helpful when you have multiple stashes saved
git stash drop stash@{number}

copy

deletes a single stash from the list of stashes
git stash clear

copy

deletes all stashes from the list, which ultimately would delete all changes

Switch

git switch name_of_branch

copy

goes to the named branch
git switch -

copy

goes back to the previous accessed branch
git switch -c name_of_branch

copy

creates a new branch and navigates to it at once

Checkout

git checkout commitHash

copy

goes to the point on time of that commit, in other words, show how the code was at the time of that commit. From there you can look around or create a new branch that will carry the code at that point.
git checkout master OR git switch master

copy

takes you back to the most recent commit
git checkout HEAD~number

copy

number will be the number of commits you want to go back in time base on where the head is at the moment. So HEAD~1 will take you back to the previous commit from where HEAD is. Note it is not from the latest commit necessarily, it is from HEAD, which can be detached at the moment.

Undo Changes

git restore fileName

copy

PERMANENTLY discards changes made in the repository (not staged or committed yet) since the last commit
git restore --staged fileName

copy

moves file from the staging area back to the working area
git reset commitHash

copy

will reset the repository back to a specific commit and move the extra commit(s) to the working tree
git reset --hard commitHash

copy

will PERMANENTLY reset the repository back to a specific commit. All the extra commits are gone!
git revert commitHash

copy

will reset the repository back to a specific commit by creating a new commit, this way history is preserved, which is a better idea when working in collaboration
